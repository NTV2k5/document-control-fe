Template Variable Table Rule Guide

Docs này giải thích từng rule cấu hình biến bảng, JSON cần viết gì, user sẽ thấy gì khi insert, và bảng render ra dạng nào.

docs/template-variable-table-rule-guide.html

Nguyên tắc: user thường không chỉnh JSON. Dev cấu hình JSON trong biến. UI đọc JSON để sinh form hỏi thêm và render table đúng trong template document.

Cách hiểu nhanh

1
controls

User sẽ thấy câu hỏi gì khi insert biến bảng, ví dụ nhập số học kỳ hoặc danh sách trường.

2
required

Key nào bắt buộc có dữ liệu trước khi render bảng.

3
render_rule

Rule engine sẽ biến table template gốc thành table cuối cùng như thế nào.

4
structure

Header, block, row template và cell config thật dùng để sinh HTML trong editor.

Rule: repeat_blocks

Dùng khi user nhập một con số, hệ thống lặp một hoặc nhiều block mẫu theo số đó.

Case đang dùng: {{tableTemplate.teaching_plan}}. User nhập số học kỳ, hệ thống sinh Học kỳ 1, Học kỳ 2, Học kỳ 3...

JSON tối thiểu

{
  "context_schema": {
    "controls": [
      {
        "type": "number",
        "key": "semesterCount",
        "label": "Số học kỳ cần sinh",
        "placeholder": "Ví dụ: 2",
        "min": 1,
        "max": 20
      }
    ],
    "required": ["semesterCount"],
    "render_rule": {
      "type": "repeat_blocks",
      "count_key": "semesterCount",
      "source_block_ids": ["semester_1_summary", "semester_1_courses"],
      "source_index": 1,
      "index_start": 1,
      "replacements": [
        { "from": "semester_1", "to": "semester_{{index}}" },
        { "from": "Học kỳ 1", "to": "Học kỳ {{index}}" }
      ]
    }
  }
}

Rule: dynamic_column_group

Dùng khi user nhập số cột con cần sinh trong một group header.

Case đang dùng: {{tableTemplate.course_plo_contribution_matrix}}. User nhập số PLO, bảng sinh cột plo_1, plo_2... 
{
  "context_schema": {
    "controls": [
      {
        "type": "number",
        "key": "ploColumnCount",
        "label": "Số cột PLO",
        "placeholder": "Ví dụ: 8"
      }
    ],
    "required": ["ploColumnCount"],
    "render_rule": {
      "type": "dynamic_column_group",
      "count_key": "ploColumnCount",
      "fixed_column_keys": ["stt", "courseCode", "courseName"],
      "plo_parent_key": "ploGroup",
      "plo_column_prefix": "plo_",
      "plo_placeholder_key": "plo_placeholder",
      "course_code_key": "courseCode",
      "course_name_key": "courseName",
      "course_code_table_field": "courses.code",
      "course_name_table_field": "courses.name"
    }
  }
}

Bản đồ nối render_rule vào structure

Các key trong rule không tự sinh độc lập. Nó phải trỏ đúng vào key thật đang có trong structure.headers, structure.blocks[].rowTemplate và cell data.

Dịch sang UI thiết kế biến bảng: trong canvas, mỗi cột hoặc nhóm header đều có một mã kỹ thuật. Rule chỉ dùng các mã đó để biết phải giữ cột nào, nhân bản cột nào, và gắn cột mới vào nhóm nào.

Cách config trên UI

Trong UI Giá trị cần đặt Nó nối vào rule nào
Câu hỏi user sẽ thấy Tên: Số cột PLO
Mã lưu dữ liệu: ploColumnCount 		count_key = ploColumnCount
Cột cố định trong canvas stt, courseCode, courseName fixed_column_keys
Nhóm header cha Tên: Chuẩn đầu ra PLO
Mã nhóm: ploGroup 		plo_parent_key = ploGroup
Cột con mẫu nằm trong nhóm PLO Tên: PLO
Mã cột: plo_placeholder
Nhóm cha: ploGroup 		plo_placeholder_key = plo_placeholder
Prefix cột sinh ra plo_ User nhập 4 thì sinh plo_1, plo_2, plo_3, plo_4
Cell chọn học phần Mã cột: courseName
Nguồn dữ liệu: courses.name 		course_name_key và course_name_table_field
Cell mã học phần tự fill Mã cột: courseCode
Nguồn dữ liệu: courses.code 		course_code_key và course_code_table_field
Cách đọc nhanh: nếu trong UI đổi Mã nhóm hoặc Mã cột, rule cũng phải đổi theo đúng mã mới. Label hiển thị có thể đổi tự do, nhưng key kỹ thuật phải khớp.

Nhiều group động: group_configs

Một số bảng không chỉ có một nhóm PLO mà có nhiều nhóm header động. Ví dụ {{tableTemplate.plo_matrix}} có 3 nhóm: knowledge, skills và competencies. Khi đó dùng render_rule.group_configs thay vì chỉ dùng một cặp plo_parent_key/plo_column_prefix. 

{
  "render_rule": {
    "type": "dynamic_column_group",
    "fixed_column_keys": ["stt", "courseName"],
    "group_configs": [
      {
        "parent_key": "knowledge",
        "parent_label": "Kiến thức",
        "column_prefix": "plo_k",
        "count_key": "knowledgePloCount",
        "label_prefix": "PLO",
        "label_start": 1,
        "group_width": 33,
        "default_count": 4
      },
      {
        "parent_key": "skills",
        "parent_label": "Kỹ năng",
        "column_prefix": "plo_s",
        "count_key": "skillsPloCount",
        "label_prefix": "PLO",
        "label_start": 5,
        "group_width": 33,
        "default_count": 4
      },
      {
        "parent_key": "competencies",
        "parent_label": "Mức tự chủ và trách nhiệm",
        "column_prefix": "plo_c",
        "count_key": "competenciesPloCount",
        "label_prefix": "PLO",
        "label_start": 9,
        "group_width": 34,
        "default_count": 4
      }
    ]
  }
}
Fallback cũng phải sạch: dù FE có normalizer, phần structure.headers trong JSON vẫn là fallback runtime. Nếu default_count là 4/4/4 thì headers tĩnh cũng chỉ nên có plo_k1..4, plo_s1..4, plo_c1..4. Không để sót headers cũ kiểu 11/11/12, vì deploy hoặc snapshot cũ có thể render thẳng fallback và gây trùng cột.
Key trong rule Phải match với cấu trúc nào Ý nghĩa khi render
count_key context_schema.controls[].key Ô input user nhập số cột cần sinh.
fixed_column_keys structure.headers[].key Những cột giữ nguyên, không sinh động.
plo_parent_key Header group có key = ploGroup và is_parent_header = true Group header cha sẽ được set lại colspan.
plo_column_prefix Không cần có sẵn đủ cột. Prefix dùng để tạo key mới. Sinh plo_1, plo_2, ...
plo_placeholder_key Header con mẫu có key = plo_placeholder và parent = ploGroup Cột mẫu bị thay bằng các cột plo_N.
course_code_key Header/cell có key courseCode trong headers, rowTemplate, rows Cột mã học phần, thường read-only sau khi chọn tên học phần.
course_name_key Header/cell có key courseName trong headers, rowTemplate, rows Cột user chọn học phần, dùng làm trigger fetch dữ liệu.
course_code_table_field rowTemplate.courseCode.table_field Nguồn DB để fill mã học phần.
course_name_table_field rowTemplate.courseName.table_field Nguồn DB để dropdown/tìm học phần.
Điểm dễ sai: nếu đổi key header trong canvas, ví dụ đổi courseName thành subjectName, thì phải đổi luôn course_name_key và cell trong rowTemplate. Nếu không, rule vẫn chạy nhưng bảng sinh ra sẽ mất dữ liệu hoặc không fetch được.

Rule: dynamic_matrix

Dùng khi user nhập số hàng và số cột. Hệ thống sinh matrix từ một block mẫu.

Case đang dùng: {{tableTemplate.syllabus_clo_plo_matrix}}. User nhập số CLO và PLO. 
{
  "context_schema": {
    "controls": [
      {
        "type": "number_pair",
        "label": "Kích thước ma trận CLO-PLO",
        "controls": [
          { "type": "number", "key": "cloCount", "label": "Số CLO" },
          { "type": "number", "key": "ploCount", "label": "Số PLO" }
        ]
      }
    ],
    "required": ["cloCount", "ploCount"],
    "render_rule": {
      "type": "dynamic_matrix",
      "row_count_key": "cloCount",
      "column_count_key": "ploCount",
      "block_id": "block_syllabus_clo_plo_matrix",
      "clo_column_key": "cloCode",
      "plo_parent_key": "ploGroup",
      "plo_column_prefix": "plo_",
      "row_id_prefix": "row_clo_"
    }
  }
}

Rule: configured_blocks

Dùng khi bảng có cấu trúc cố định theo nhiều block, nhưng block nào là title, summary, courses được config trong JSON.

Case đang dùng: {{tableTemplate.curriculum_framework_fixed}}. Dòng summary tự cộng số TC, LT, TH từ các block môn học bên dưới. 
{
  "render_rule": {
    "type": "configured_blocks",
    "field_map": {
      "stt": "stt",
      "course_code": "courseCode",
      "course_name": "courseName",
      "credits": "credits",
      "lecture": "lecture",
      "practice": "practice"
    },
    "block_configs": [
      { "kind": "title", "id": "framework_title", "label": "Khung chương trình đào tạo" },
      {
        "kind": "summary",
        "id": "sum_1_1",
        "label": "1.1. Lý luận chính trị",
        "source_course_block_ids": ["rows_1_1"]
      },
      { "kind": "courses", "id": "rows_1_1", "default_required_type": "BB" }
    ],
    "title_merge_colspan": 8,
    "summary_merge_colspan": 3
  }
}

Behavior: manual và computed field

structure.behavior điều khiển cell nào user được nhập tay và cell nào tự tính. 

{
  "structure": {
    "behavior": {
      "manual_fields": ["credits", "theoryHours", "practiceHours", "integratedHours"],
      "editable_read_only_fields": ["totalCredit"],
      "computed_fields": [
        {
          "target_field": "totalHours",
          "source_fields": ["theoryHours", "practiceHours", "integratedHours"],
          "type": "sum"
        }
      ]
    }
  }
}
Key Dùng để làm gì
manual_fields Không auto dropdown theo table_field, cho user nhập trực tiếp.
editable_read_only_fields Cell đang read-only trong fixture nhưng UI vẫn cho sửa.
computed_fields Cell tự tính từ field khác, ví dụ tổng LT + TH + tích hợp.

Document template variables

Biến documentTemplate.* render một block HTML có cấu trúc sẵn. Nhóm này dùng cho các đoạn như thông tin chung học phần, mô tả học phần, mục tiêu, tài liệu tham khảo, yêu cầu sinh viên.

Loại key Ví dụ Cách dùng
Key mới dạng namespace documentTemplate.syllabus.general_info
documentTemplate.syllabus.course_description 		Mỗi key là một section riêng. Dùng bộ này khi file template muốn tự sắp thứ tự các mục và tránh render trùng nội dung.
Key legacy dạng flat documentTemplate.syllabus_general_info Có thể là block lớn đã chứa nhiều mục con. Không trộn cùng các section tách nhỏ nếu nội dung bị lặp.
Header dùng một lần documentTemplate.national_header_no_border Chỉ đặt ở đầu file. Nếu đặt thêm ở cuối file, preview sẽ render thêm header dư phía dưới.
Case dễ bị trùng: documentTemplate.syllabus_general_info cũ có thể đã chứa mục 2. Mô tả học phần. Nếu template cũng chèn thêm documentTemplate.syllabus.course_description thì preview sẽ thấy hai mục mô tả học phần. Cách đúng là dùng documentTemplate.syllabus.general_info cho mục thông tin chung, rồi chèn riêng documentTemplate.syllabus.course_description.
Khi copy biến từ file cũ: rà lại key hiện tại trong JSON definitions trước khi paste. Nếu thấy key cũ không render hoặc render thừa section, thay bằng key mới trong cùng nhóm documentTemplate.syllabus.*.

Snapshot/live config

Template variable có hai nguồn cấu hình cần phân biệt rõ: JSON definition hiện tại và snapshot đã lưu trong template/document.

Nguồn Dùng khi nào Lưu ý
JSON definition trong document-management-be/src/modules/templates/template-variables/template-variable-definitions Biến mới, reset biến, hoặc chế độ live config. Đây là canonical config cho UI và render engine.
Snapshot trong template.variables hoặc document.data Template/document đã lưu trước đó, đặc biệt bản đã publish. Snapshot bảo toàn dữ liệu cũ. Sửa JSON không tự phá các bản đã lưu nếu đang ở chế độ snapshot.
Setting TEMPLATE_VARIABLE_RENDER_SETTINGS Chọn render theo snapshot hay dùng live config cho draft. Dùng live config để draft bắt cấu hình mới; published nên giữ snapshot để tránh đổi layout ngoài ý muốn.
Khi sửa JSON mà UI vẫn render kiểu cũ: kiểm tra biến đang dùng snapshot hay live config. Nếu cần lấy config mới cho một template/document cụ thể, reset/chọn lại biến hoặc bật live config cho draft rồi lưu lại.

Deploy checks

Khi local đúng nhưng deploy render khác, kiểm tra theo thứ tự dưới đây trước khi debug UI.

Check Lệnh/API gợi ý Kỳ vọng
BE đã build lại JSON definition cd document-management-be && yarn build File JSON trong dist/src/.../template-variable-definitions giống source.
API deploy trả đúng config mới GET /api/v1/templates/template-variables/template-tables-plo-matrix context_schema.render_rule.type là dynamic_column_group và fallback headers của plo_matrix là 4/4/4.
FE deploy đã có normalizer mới cd document-manager && yarn build UI hiển thị panel nhập Số PLO kiến thức, Số PLO kỹ năng, Số PLO tự chủ/trách nhiệm.
Data đang xem không giữ snapshot cũ Kiểm tra template.variables.template_structures hoặc document.data.template_structures. Nếu snapshot còn headers cũ, reset/chọn lại biến hoặc bật live config cho draft.
Case plo_matrix: local đúng nhưng deploy bung 11/12 cột thường là do deploy đang đọc fallback cũ hoặc snapshot cũ. Sửa fallback JSON về đúng default count và build lại BE/FE trước, sau đó xử lý snapshot nếu template/document đã lưu từ trước.

Checklist chống lỗi render

Check Lý do
controls[].key khớp required[] Nếu lệch key, insert sẽ báo thiếu config dù user đã nhập.
render_rule.count_key khớp control number key Rule không đọc được số để sinh block/cột.
source_block_ids tồn tại trong structure.blocks repeat_blocks không tìm được block mẫu.
Header parent/child khớp key Nếu parent sai, bảng bị lệch header hoặc mất colspan.
table_field đúng dạng table.field Dropdown và auto fetch row data mới lấy đúng nguồn.
Cell read-only cần sửa thì thêm vào editable_read_only_fields Tránh case user không nhập được số hoặc STT.
Field tự tính thì config computed_fields Không hardcode logic tính tổng trong component.
Khi nào mới cần code mới? Chỉ khi behavior không thể biểu diễn bằng 5 rule hiện tại. Khi đó thêm rule type generic mới vào rule engine, không viết theo tên biến cụ thể.
